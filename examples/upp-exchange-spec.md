# Спецификация обмена с 1С:УПП

Этот документ описывает рекомендуемый формат выгрузки из `1С:УПП` в дашборд.

## Цель

Передавать один пакет данных за период в endpoint:

`POST /api/ingest/upp`

Пакет должен быть идемпотентным:

- один и тот же `packageId` не должен отправляться как новый импорт
- одинаковый payload отсекается по `payloadHash`

## Заголовки

- `Content-Type: application/json`
- `X-API-Key: <секретный ключ>`

## Рекомендуемый сценарий из УПП

1. Регламентное задание формирует данные за нужный период.
2. УПП собирает единый JSON-пакет.
3. УПП выполняет `HTTP POST` в дашборд.
4. При ответе `status=success` пакет считается принятым.
5. При ответе `status=duplicate` повторно отправлять тот же пакет не нужно.
6. При ошибке пакет должен быть записан в журнал УПП и поставлен на повторную отправку.

## Формат пакета

```json
{
  "sourceSystem": "1c-upp",
  "sourceObject": "regulated_exchange",
  "packageId": "upp-2026-04-22-001",
  "period": "2026-04",
  "stores": [
    { "id": "irk-1", "name": "Иркутск Центр", "region": "Иркутск" }
  ],
  "products": [
    { "id": "coffee", "name": "Кофе", "category": "Напитки" }
  ],
  "plans": [
    { "storeId": "irk-1", "productId": "coffee", "amount": 120000 }
  ],
  "sales": [
    {
      "storeId": "irk-1",
      "productId": "coffee",
      "amount": 12500,
      "cost": 7200,
      "quantity": 48,
      "soldAt": "2026-04-22T09:00:00+08:00"
    }
  ],
  "marketingMetrics": [
    {
      "channelId": "yandex-direct",
      "channelName": "Яндекс Директ",
      "spend": 85000,
      "leads": 620,
      "orders": 164,
      "revenue": 402000,
      "impressions": 480000,
      "clicks": 11800,
      "sessions": 9700
    }
  ]
}
```

## Правила маппинга из УПП

### `stores`

Использовать:

- код торговой точки, подразделения или склада как `id`
- человекочитаемое наименование как `name`
- город/регион как `region`

### `products`

Использовать:

- код номенклатуры как `id`
- наименование номенклатуры как `name`
- группу номенклатуры как `category`

### `plans`

Каждая строка:

- `storeId`
- `productId`
- `amount` в рублях

Если в УПП план хранится в разрезе только точки без товара, рекомендуется:

- либо раскладывать по товарным группам заранее
- либо передавать искусственную товарную группу, например `all-products`

### `sales`

Каждая строка:

- `storeId`
- `productId`
- `amount` — выручка
- `cost` — себестоимость, если доступна
- `quantity`
- `soldAt` — дата/время продажи

Рекомендуется передавать уже очищенные продажи:

- без служебных документов
- без дублей
- с учетом возвратов по согласованному правилу

### `marketingMetrics`

Если маркетинг не живет в УПП, блок можно:

- не передавать вообще
- или формировать из отдельного промежуточного источника

## Ответ сервиса

### Успех

```json
{
  "ok": true,
  "run": {
    "status": "success"
  }
}
```

### Дубликат

```json
{
  "ok": true,
  "run": {
    "status": "duplicate"
  }
}
```

### Ошибка

```json
{
  "error": "Описание ошибки",
  "run": {
    "status": "failed"
  }
}
```

## Рекомендации по эксплуатации

- Формируйте `packageId` детерминированно, например `upp-YYYY-MM-DD-NNN`
- Отправляйте пакет за период целиком, а не по одной строке
- Для продаж используйте полную замену периода в рамках пакета
- При ошибке сохраняйте JSON пакета в журнале УПП
- Повторную отправку делайте тем же `packageId`

## Что лучше уточнить в УПП перед внедрением

- где в конфигурации лежит план продаж
- какой документ или регистр считать фактом продаж
- как учитывать возвраты
- откуда брать себестоимость
- как кодируются точки и номенклатура
- нужен ли разрез по менеджерам, подразделениям и каналам
